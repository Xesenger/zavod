CONTROL CLASSES
1. Read-only projection controls
- show status
- open docs
- reveal evidence

2. Surface-navigation controls
- move between visible Work Cycle surfaces
- return to result
- return to discussion surface
- these do not necessarily mutate backend/core truth
- IMPORTANT:
  - temporary recovery-only UI buttons may exist here
  - they are not automatically part of the final product model
  - if a button exists only to help recovery navigation or visual inspection, it must be treated as temporary unless re-proven from product semantics

3. Core-driving controls
- affect lifecycle/runtime/result truth
- must be reconnected only after backend owner is proven

WORK CYCLE CONTROL GROUPS

GROUP A. CHAT PROJECTION
- Kind:
  - read-only at first
- Examples:
  - discussion summary
  - phase text
  - frozen context projection
- Risk:
  - medium if it starts inventing intent readiness

GROUP B. CHAT / AGREEMENT ACTIONS
- Kind:
  - core-driving
- Examples:
  - send
  - continue
  - start validation/preflight
- Risk:
  - high
- Rule:
  - do not reconnect until intent/phase ownership is proven
- Important:
  - returning from validation/preflight back into discussion must not be assumed to leave the current step untouched
  - later discussion may refine, replace, or collapse the current step path depending on core truth

GROUP C. EXECUTION PROJECTION
- Kind:
  - read-only at first
- Examples:
  - stage labels
  - runtime/evidence labels
  - document links
- Risk:
  - low if truthful

GROUP D. EXECUTION ACTIONS
- Kind:
  - core-driving
- Examples:
  - confirm preflight
  - clarify preflight
  - cancel execution
  - resume execution
- Risk:
  - high
- Important:
  - revision mode also needs its own input path inside the same Execution surface
  - revision is not just a static status label

GROUP E. RESULT PROJECTION
- Kind:
  - read-only at first
- Examples:
  - result summary
  - review status
  - metadata/evidence projection
- Risk:
  - low if truthful

GROUP F. RESULT DECISIONS
- Kind:
  - core-driving
- Examples:
  - accept
  - revise / return for revision
  - reject / abandon style decision
- Risk:
  - very high
- Hard rule:
  - repo mutation only through Accept

GROUP G. WORK CYCLE SURFACE NAVIGATION
- Kind:
  - surface-navigation
- Examples:
  - return to result
  - return to execution surface
  - return to discussion context
- Risk:
  - low to medium
- Rule:
  - must not be assumed to mutate backend truth automatically
- Additional rule:
  - if these buttons were introduced only during recovery to help navigate currently visible surfaces,
    they should be treated as temporary recovery scaffolding, not as confirmed product controls

RETURN-SEMANTICS RULE
- Do not collapse all "return" buttons into one meaning.
- Two different families exist:
  1. shell/surface navigation returns
     - purely UI-level
     - do not mutate truth
  2. lifecycle returns
     - may re-open discussion
     - may preserve active truth
     - may alter the current step path
     - may remove the current step from the active visible path
- Therefore labels such as:
  - return to chat
  - return to lead
  - back from preflight
  - return to result
  must always be mapped to their exact backend/core transition before wiring

PROTOTYPE RULE
- Early UI prototypes may be used as visual references for:
  - proportions
  - column layout
  - lightness / density
- They must not be treated as authoritative control truth by themselves
- Product/control truth still requires confirmation from:
  - canon
  - current code
  - explicit model decisions

SAFE RECONNECTION ORDER
1. Read-only projection controls
2. Surface-navigation controls
3. Chat/agreement controls
4. Execution controls
5. Result decisions

DO NOT DO
- Do not assume every button inside Work Cycle is a backend command.
- Do not assume every Result button writes to repo.
- Do not connect controls from old recovery docs without re-proving ownership from current code.
