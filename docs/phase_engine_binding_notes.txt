ZAVOD PROJECT WORK CYCLE PHASE ENGINE BINDING NOTES
Date: 2026-04-11
Scope: Project Work Cycle only

IMPORTANT
- This file is about binding the existing phase engine to Project Work Cycle UI.
- It is not a whole-app navigation document.
- Whole-app model starts from:
  - Chats
  - Projects
- App-level UI model reference:
  - docs/ui_product_model.txt

KEY FINDING
- A real phase engine already exists in code.
- Project Work Cycle should bind to that engine instead of inventing a local one.

PRIMARY EXISTING TYPES
Source: Flow/StepPhaseContracts.cs

1. SurfacePhase
- Discussion
- Execution
- Result
- Completed

2. DiscussionSubphase
- Idle
- Forming
- Ready
- Reopened

3. ExecutionSubphase
- Preflight
- Running
- Qc
- Revision
- Interrupted

4. ResultSubphase
- Ready
- RevisionRequested

5. StepPhaseState
- current phase and subphase truth for Work Cycle projection

6. StepPhaseProjection
- surface visibility and allowed action projection

WHY THIS MATTERS
- Work Cycle UI does not need a fake local lifecycle model.
- The existing engine already expresses:
  - discussion
  - preflight
  - running
  - QC
  - result
  - revision
  - interrupted

CURRENT BINDING RULE
- Project Work Cycle should eventually consume:
  - StepPhaseState
  - StepPhaseProjection
- Then bind:
  - ShowChat -> Work Cycle Chat surface
  - ShowExecution -> Execution surface
  - ShowResult -> Result surface

ACTION BINDING RULE
- Projection can gate backend-driving controls such as:
  - send/start-entry
  - preflight confirm/clarify
  - cancel/resume execution
  - accept
  - return for revision
  - return to lead/discussion
- But not every visible button is backend-driving.
- Surface-navigation buttons may still exist outside projection-owned backend actions.

IMPORTANT DISTINCTION
- Top-level Projects navigation:
  - List
  - Home
  - Work Cycle
  is not owned by StepPhaseProjection.
- StepPhaseProjection applies only inside Work Cycle.

RESUME RULE
- ResumeStageStorage and ResumeStageNormalizer already implement degraded honest restore.
- Therefore Work Cycle binding should prefer normalized resume input over UI-local memory.

BACKEND OWNERS ALREADY PRESENT
- discussion/execution/result phase engine:
  - Flow/*
- resume normalization:
  - Persistence/ResumeStageStorage
  - Persistence/ResumeStageNormalizer
- result/apply/abandon:
  - State/*
  - Boundary/*
- closure proposal support:
  - Execution/*

DO NOT DO
- Do not invent a second Work Cycle phase enum in MainWindow.
- Do not use phase engine as whole-app navigation state.
- Do not use old recovery booleans instead of StepPhaseProjection where current code already has the real concept.

MINIMAL SAFE TARGET
- When Work Cycle returns, the minimum binding target should be:
  - StepPhaseState phaseState
  - StepPhaseProjection projection
- Then:
  - Chat / Execution / Result visibility follows projection
  - Frozen/stubbed styling follows subphase and action class
