SAFE WORK CYCLE PHASE MODEL
Phase A. Discussion Entry
- Visible:
  - Chat
- Hidden:
  - Execution
  - Result
- Truth basis:
  - no active work, or honest discussion-first project state

Phase B. Agreement / Preflight
- Visible:
  - Chat
  - Execution as prepared shell
- Hidden:
  - Result
- Truth basis:
  - discussion reached validation/preflight readiness

Phase C. Active Work
- Visible:
  - Chat frozen
  - Execution active
- Hidden:
  - Result
- Truth basis:
  - active shift/task truth with work context

Phase D. Result Ready
- Visible:
  - Chat frozen
  - Execution visible
  - Result active
- Truth basis:
  - result/review support is genuinely present

Phase E. Revision Intake
- Visible:
  - Chat frozen
  - Execution active
  - Result visible
- Truth basis:
  - task remains active and revision path exists

Phase F. Post-Decision Return
- Visible:
  - Chat
- Hidden:
  - Execution
  - Result
- Truth basis:
  - decision cycle resolved
- Important:
  - hidden execution/result does not itself prove shift closure

PHASE ENGINE RULE
- Do not invent a new local Work Cycle phase model.
- Use:
  - StepPhaseState
  - StepPhaseProjection
- Work Cycle UI must project the existing engine, not replace it.

WORK CYCLE SURFACE ROLES
1. Chat
- Role:
  - discussion
  - agreement/preflight
  - frozen context during work/result
- Must not:
  - invent validated intent
  - override active truth
  - behave like free-chat mode

2. Execution
- Role:
  - persistent work surface
  - active work
  - active again during revision
- Must not:
  - fake live running after restart
  - replace task truth
  - collapse into Result

3. Result
- Role:
  - decision surface
  - accept / revise / reject-or-abandon style decisions
- Must not:
  - imply completion just by visibility
  - replace Execution
  - delete history

BUTTON CATEGORY RULE
- Not every Work Cycle button mutates backend/core.
- Two button categories must stay separate:
  - core-driving actions
  - UI/surface navigation actions
- Example of non-core button:
  - return to result

REPO MUTATION RULE
- Real repo mutation happens only through:
  - Accept
  - in Result
- Not through navigation.
- Not through entering Work Cycle.
- Not through showing Execution.

SAFE REBUILD ORDER FOR WORK CYCLE
1. Restore phase-driven shell only

3. Restore read-only Execution projection
4. Restore read-only Result projection
5. Restore non-mutating navigation controls

HARD WORK CYCLE RULES
- Do not treat Work Cycle as the whole app.
- Do not treat Work Cycle Chat as top-level Chats mode.
- Do not let Work Cycle navigation mutate truth.
- Do not write repo changes before Accept.
- Do not rebuild the full heavy shell in one pass.
