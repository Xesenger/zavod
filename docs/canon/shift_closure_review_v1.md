# ZAVOD v1 — Shift Closure Review

## Purpose

`Shift Closure Review v1`

- defines the review boundary before a shift becomes `completed`
- introduces the explicit user-confirmed closure step before historical fixation
- aligns closure summary, project updates, and snapshot creation under one closure path
- preserves closure as the only production truth-update wall

---

## Definition

`Shift Closure Review` is the final review step between:

- active shift/work state
and
- completed historical shift state

It exists to ensure that closure is:

- explicit
- reviewed
- minimally grounded in shift evidence
- confirmed by the user before final apply

It is not:

- task execution
- runtime QC
- snapshot itself
- arbitrary summary generation without confirmation

---

## Position in Architecture

Shift Closure Review happens only after active work inside the shift has reached a closable state.

It sits between:

- active shift lifecycle
and
- closure apply / snapshot / archive history

Formula:

active shift
→ closure review
→ closure apply
→ snapshot/history
→ shift completed

---

## Core Principle

Closure review is the last explicit human-facing review wall before the shift leaves `active`.

This preserves:

- honesty of summary
- correctness of project-level updates
- explicit user confirmation
- consistency between closure outcome and snapshot/history

Formula:

review before finalize

---

## Trigger

Closure review begins when the user/system initiates:

- `Close Shift`

This trigger is only valid when closure is actually allowed by adjacent lifecycle rules.

Closing a shift must not happen silently.

---

## Canonical Flow

1. user initiates `Close Shift`
2. lead layer builds a proposed closure summary
3. the proposed summary is shown to the user
4. user may:
   - confirm as-is
   - provide optional clarification
   - cancel closure
5. if confirmed:
   - system builds final closure payload
   - system applies project updates through closure path
   - system creates snapshot/history artifacts
   - shift becomes `completed`

This is the canonical v1 closure path.

---

## Proposed Closure Summary

The system must first produce a proposed closure summary.

This summary is not yet final project truth.
It is the reviewable candidate closure description.

The summary is generated by the lead layer,
not by raw worker output and not by UI-local state.

---

## Summary Structure

The proposed closure summary should briefly reflect:

- what was worked on
- what was completed
- what changed, if anything changed
- what decisions were made, if any
- what requires project-level update, if applicable
- what remains open, if anything remains open

The summary must stay concise and closure-oriented.
It is not a full replay of the shift.

---

## Sources

Closure review summary must be grounded in shift evidence.

Allowed closure inputs include:

- active shift/task truth
- accepted results
- relevant decisions
- unresolved items
- closure-ready evidence from the shift
- bounded trace/evidence used only as support

Forbidden summary sources:

- full raw chat replay
- UI-local memory
- arbitrary model memory
- purely aesthetic summary generation detached from actual shift evidence

---

## User Clarification

The user may add an optional clarification before final closure.

Rules:

- clarification is optional
- clarification does not replace the whole summary
- clarification refines, corrects, or supplements the interpretation of the shift
- clarification becomes part of closure evidence for the final payload

This keeps the user inside the closure wall
without forcing a second full drafting pass.

---

## Cancel Path

If the user cancels closure review:

- the shift remains `active`
- no closure apply occurs
- no snapshot is created
- no project truth is updated through closure

Cancel is therefore a valid non-closing path.

---

## Confirmation Rule

A shift may leave `active` only after explicit user confirmation of closure review.

Confirmation authorizes:

- final closure payload assembly
- closure apply path
- snapshot creation
- shift completion

Without confirmation:

- the shift must remain active
- closure must not be finalized

---

## Final Closure Payload

After confirmation, the system assembles a final closure payload.

This payload may include:

- finalized closure summary
- accepted outcomes
- relevant decisions
- unresolved items
- project-update implications
- user clarification if one was provided

The closure payload is the structured basis for:

- project update apply
- snapshot creation
- shift completion

---

## Apply Rule

After confirmation, closure applies only the justified updates.

Rule:

- update only those project layers for which the shift provides actual basis

Consequences:

- no automatic broad rewrite of project docs
- no speculative updates to canon/roadmap/direction
- no project-level mutation without closure evidence

This keeps closure minimal and grounded.

---

## Relation to Project Truth

Closure review itself does not mutate project truth.

It prepares the reviewed closure payload.

Project-level updates happen only after confirmation,
through the closure apply path.

Formula:

review proposes
closure apply mutates

---

## Relation to Snapshot

Snapshot is created after confirmed closure,
not instead of closure review.

Relationship:

- closure review validates the final closure framing
- closure apply finalizes update path
- snapshot records the historical closure outcome

Formula:

closure review confirms
snapshot records

---

## Relation to Shift Lifecycle

Shift lifecycle only knows:

- `active`
- `completed`

Shift Closure Review is the human-facing review wall before the transition:

- `active -> completed`

The lifecycle transition itself happens only after confirmed closure apply.

---

## Canons

- shift closure requires explicit user confirmation
- proposed closure summary is reviewable and not yet final truth
- optional user clarification becomes part of closure evidence
- closure review is led by the lead layer
- closure review itself does not mutate project truth
- only closure apply may finalize project updates
- only justified project layers may be updated through closure
- snapshot is created after confirmed closure, not before
- cancelling closure leaves the shift active

---

## Boundaries

This document does not define:

- task execution
- runtime QC
- full trace implementation
- snapshot structure internals
- archive storage internals
- multi-user approval chains
- repeated confirmation ladders
- complex negotiation workflows

These belong to adjacent canon documents.

---

## Exclusions

Excluded from `Shift Closure Review v1`:

- automatic shift closure
- silent closure without user confirmation
- full manual editing of shift history
- repeated mandatory multi-step confirmation flows
- arbitrary regeneration loops of the closure summary
- broad speculative project-document rewrite during closure

---

## Operational Notes

- closure is the point where project truth may finally be updated
- the system should update only the minimally justified set of documents
- if no explicit shift-backed decision/change exists,
  foundational documents such as canon/roadmap/direction should remain untouched
- the system does not need to re-render the final post-clarification payload before apply in v1
