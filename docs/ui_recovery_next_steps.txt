ZAVOD UI RECOVERY / REBUILD PLAN
Date: 2026-04-11
Status: active replacement for the older recovery sequence

IMPORTANT
- This file replaces the older recovery plan that started from the wrong abstraction level.
- Treat older recovery notes with distrust unless they are re-confirmed by current code.
- Product truth remains in docs/zavod/canon/.
- Code remains the source of current implementation reality.
- App-level UI model reference:
  - docs/ui_product_model.txt
- Current verification checkpoint:
  - docs/ui_verification_checkpoint_2026-04-12.txt

CORE CORRECTION
- The app must not be rebuilt from the inner 3-surface shell outward.
- The app must be rebuilt from the top-level product split first:
  - Chats
  - Projects

TOP-LEVEL PRODUCT MODEL
1. Chats
- Free chats.
- Not part of ZAVOD project lifecycle.
- No project truth ownership.
- No shift/task/result lifecycle.
- No Three-Part Prompt Assembly & Anchor System requirement.
- No Execution Streaming & Chat Projection Layer requirement.

2. Projects
- This is the actual entry into ZAVOD.
- Projects mode contains three internal UI levels:
  - Project List
  - Project Home
  - Project Work Cycle

PROJECTS INTERNAL MODEL
1. Project List
- Contains:
  - import existing project
  - create new project
  - choose existing project
- Navigation only.
- Must not mutate project truth by itself.

2. Project Home
- First implementation should stay intentionally simple.
- Initial contents:
  - open HTML summary in external browser
  - scanner/import material list
  - simple status lines
- This is not the work cycle.
- This is not the result surface.
- This is not execution control.

3. Project Work Cycle
- This is the inner ZAVOD core.
- Use the UI v2 phase concept as the working shape.
- Practical internal surfaces remain:
  - Chat
  - Execution
  - Result
- Prompt assembly, anchors, execution streaming, and chat projection belong only here.

IMPORTANT DISTINCTION
- Chat surface inside Project Work Cycle != top-level Chats mode.
- Top-level Chats mode is free chat.
- Project Work Cycle chat is bounded project workflow UI.

PROJECT STAGE MODEL
- Current practical project progression:
  - Import
  - Pre-canon docs
  - Canon
- This progression belongs to Projects mode, not to top-level Chats.

PROJECT HOME V1 RULE
- Open generated HTML through external browser for now.
- WebView2 can be added later.
- Do not block recovery on embedded web rendering.

SCANNER / MATERIALS RULE
- Scanner must remain broad and evidence-oriented.
- Project Home should show user materials found by scanner/import pipeline, including:
  - pdf
  - txt
  - md
  - archives
  - images
  - office-like materials
  - other supported evidence payloads
- These are evidence/material surfaces, not canon truth by themselves.

NAVIGATION RULE
- Navigation inside Projects:
  - Project List -> Project Home -> Project Work Cycle
  - back from Work Cycle to Home
  - back from Home to List
- This changes UI state only.
- Navigation must not mutate project truth by itself.

REPO MUTATION RULE
- Real repo changes appear only through:
  - Accept
  - in Result
- Not through navigation.
- Not through entering Work Cycle.
- Not through preview docs alone.
- Not through execution visibility.

COLD START RULE
- Cold start is valid.
- Empty state must not confuse or break ZAVOD.
- In future, Cold Start also includes:
  - New Project entry
  - user arrives without a clear project yet
  - environment preparation may happen only with explicit user consent
- For current rebuild:
  - keep cold start honest
  - do not fake active work
  - do not fake remembered continuity

WORK CYCLE BUTTON RULE
- Not every button inside Project Work Cycle calls backend/core transitions.
- There are two categories:
  - core-driving actions
  - UI/surface navigation actions
- Example of non-core action:
  - return to result
- Therefore button reconstruction must not assume every visible control mutates lifecycle truth.

REBUILD ORDER
Step 1. Preserve runnable host
- Keep the app launching reliably.
- Do not re-break startup/package recovery.

Step 2. Restore top-level app shell
- Restore the app-level split:
  - Chats
  - Projects
- Restore central title-bar switch.
- Restore mode-specific visual identity:
  - Chats = light / Lora
  - Projects = anthracite / IBM Plex Sans

Step 3. Restore Chats mode as a separate host
- Use ChatsAdapter.
- Restore only free chat shell behavior.
- Keep it independent from project lifecycle.

Step 4. Restore Projects root shell
- Add internal navigation state for:
  - Project List
  - Project Home
  - Project Work Cycle
- Do not jump directly into the 3-surface workflow.

Step 5. Restore Project List
- Show import/new/open entry actions.
- Keep it navigation-only at first.

Step 6. Restore Project Home v1
- Add:
  - open HTML summary button
  - materials list
  - simple statuses
- Keep Home light and read-only.

Step 7. Restore Project Work Cycle shell
- Use the UI v2 phase concept as the working model.
- Bind the shell to the existing phase engine already present in code.
- Restore only visibility and read-only projection first.

Step 8. Restore Work Cycle content in safe order
- Chat projection first
- Execution projection second
- Result projection third
- All mutation controls remain disabled initially.

Step 9. Restore controls by category
- First:
  - non-mutating navigation
  - open docs
  - return to result / return to surface
- Later:
  - discussion/composer
  - execution controls
  - result decisions

Step 10. Reconnect backend-mutating actions one by one
- Only after the UI shell remains stable across launches.
- Only after the action owner is proven from code.

DISTRUST RULE FOR OLDER RECOVERY DOCS
- Older docs may still contain useful fragments.
- But they may also:
  - start from the wrong top-level model
  - treat Chats/Projects as shell variants instead of product entry modes
  - assume the 3-surface shell is the app root
- Therefore:
  - use them as hints only
  - verify against canon + current code before reuse

HARD DO-NOT-DO LIST
- Do not rebuild from the 3-surface shell outward.
- Do not treat top-level Chats and Project Work Cycle chat as the same thing.
- Do not let navigation mutate truth.
- Do not write to repo before Accept.
- Do not reintroduce the previous huge MainWindow in one pass.
- Do not trust old recovery assumptions without code confirmation.

SUCCESS CRITERIA
- The app launches.
- Chats mode works as free chat.
- Projects mode works as ZAVOD entry.
- Project List, Project Home, and Project Work Cycle are separated cleanly.
- Project Home stays simple.
- Work Cycle uses bounded project semantics.
- Repo mutation happens only through Accept in Result.
