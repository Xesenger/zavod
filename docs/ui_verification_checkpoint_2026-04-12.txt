CONFIRMED TOP-LEVEL MODEL
1. App modes
- Chats
- Projects

2. Projects internal navigation
- Project List
- Project Home
- Project Work Cycle

3. Navigation rule
- Shell navigation is UI-only.
- List -> Home -> Work Cycle does not mutate project truth by itself.

4. Repo mutation rule
- Real repo changes appear only through Accept in Result.

CONFIRMED WORK CYCLE MODEL
1. Three surfaces
- Chat
- Execution
- Result

2. Scope
- This 3-phase UI exists only inside:
  - Projects
  - Project Work Cycle
- It does not describe top-level Chats mode.

3. Phase-driven visibility
- Discussion entry:
  - Chat visible
  - Execution hidden
  - Result hidden
- Agreement / Preflight:
  - Chat visible
  - Execution visible
  - Result hidden
- Active work:
  - Chat frozen
  - Execution active
  - Result hidden
- Result ready:
  - Chat frozen
  - Execution visible
  - Result active
- Revision:
  - Chat frozen
  - Execution active again
  - Result remains visible

CONFIRMED EXECUTION SEMANTICS
1. Execution is one persistent surface
- Validation/preflight is not a separate second window.
- Running is not a replacement window.
- Revision returns to the same persistent Execution surface.

2. Work entry
- Send:
  - sends a discussion message only
  - does not open Execution by itself
- "В работу" / work-entry action:
  - is separate from Send
  - opens Execution when readiness allows it

3. Validation / preflight mode inside Execution
- Shows the current agreement.
- Expected buttons by semantics:
  - Confirm
  - Fix / Clarify
  - Return to chat

4. Confirm in validation / preflight
- Does not open a new window.
- Switches the same Execution surface into active work mode.

5. Fix / Clarify in validation / preflight
- Opens clarification with the lead.
- Does not necessarily mean a full return to the large Chat surface.

6. Result appearance
- Result appears automatically after work is done.
- Result does not replace Execution.

7. Revision
- "На доработку" returns to the same Execution surface in revision mode.
- Revision should include input inside that same Execution surface.

KNOWN DANGEROUS AREAS
1. Return-style controls
- They must not be treated as harmless navigation by default.
- Current core already shows different consequences for:
  - CancelPreflight
  - ReturnToLead
  - OpenInterruptedDiscussion
  - ReturnToResultFromRevisionIntake

2. Pause semantics
- "Pause" is not yet clearly modelled as a distinct product truth in current core.
- Current code is closer to:
  - cancel/interrupted
  - resume
- Therefore pause must not be invented casually in UI.

3. Discussion/composer boundary
- This is the next risky layer.
- It must not be reintroduced as if it were harmless local chat.
- It needs a real owner path for:
  - discussion send
  - work-entry readiness
  - validation clarification
  - revision clarification

PROTOTYPE RULE

  - broad visual direction
- They are not authoritative lifecycle/control truth by themselves.
